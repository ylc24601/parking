## 一、系統怎麼運作（整體邏輯）

> 完整規格見 [Church_Parking_Management_System_PRD.md](Church_Parking_Management_System_PRD.md)；以下是給 admin 看的白話版。

### 1.1 車位怎麼算

地下室固定 **23 個車位**，但每週想停車的人可能有 40–50 台車，所以系統每週都要算出「這週實際能開放給一般會友申請的車位數」：

```
可分配車位 = 總車位（23）− 保留／停用車位 − 全職同工本週保留數
```

- **保留／停用車位**：外賓保留位＋當週故障或停用的車位，兩者合併成一個數字，在 [車位設定](#車位設定-admincapacity) 頁操作。
- **全職同工保留數**：全職同工（P1）名單裡，扣掉本週自己標記「不停車」的人——這個數字系統自動算，**Admin 改不動**。
- 這個公式**由系統原子計算並鎖住**：如果你想改的數字會讓「可分配」小於「本週已經核准的人數」，系統會直接拒絕（不會讓車位變成負的、超賣）。

### 1.2 誰優先停車（排序規則）

車位不是先搶先贏，而是照「關懷優先」排序：

| 等級 | 對象 | 說明 |
|---|---|---|
| **P1** | 全職同工／外賓 | 完全不進入排序，開放前就先扣掉名額 |
| **P2**（關懷優先） | 行動不便、孕婦、年長者同行、幼兒同行 | 需 Admin 核准過的資格；同一批 P2 之間再依「多久沒停到車」排序 |
| **P3**（一般會友） | 其餘所有申請人 | 先看違規分數（0–3，分數低者優先），再看多久沒停到車 |

P2 資格的核准／撤銷／到期追蹤走 [資格審查](#資格審查-admineligibility) 頁；每一種 P2 類型的細節（要不要每週宣告、效期多久）見 PRD 第五節。

### 1.3 一週的時間軸

```
週一 09:00 ── 開放本週申請（所有申請先進 pending 排隊）
週五 18:00 ── 系統自動分配：依排序規則核准，額滿的人進候補
              （之後仍開放「晚鳥」申請，隨到隨核准直到滿額）
週六 ────── 有人取消 → 候補依序遞補：
              18:00–23:59 取消 → 遞補者收到 LINE 通知，2 小時內要確認，
                                逾時或放棄則換下一位（原候補排位不變）
              週日 00:00 後取消 → 直接核准遞補者，不用等確認
主日 10:00–10:30 ── 現場點名開始
     10:20 ── 已核准但未到場的 P2 會友收到提醒通知（保留到 10:45）
     10:30 ── 一般會友（P3）未到場者，保留車位自動釋出；
              同時廣播通知候補者可到現場詢問同工
     10:45 ── P2 會友未到場（且沒回「正在路上」）者，保留車位釋出
              （回過「正在路上」的可延到 10:55）
              ── 10:30 後現場調度優先，不再保證依候補順位保留車位
Staff 結束點名 ── 整場沒出現、也沒取消的預約，結算為 no_show
```

### 1.4 取消不一定算違規

- **週五 18:00 前取消**：完全無記錄。
- **週五 18:00 之後、主日 10:30 前取消**：不計違規分數（系統理解是臨時狀況），並立刻觸發候補遞補。

### 1.5 No-show（完全沒來）的後果

- **一般會友（P3）**：沒來一次，違規分數 +1（上限 3 分），下次排序會往後；只要之後成功停車一次，分數就 −1。
- **P2 會友**：沒來**不扣分**（恩慈防護網），但系統會在 Admin 後台開一個「牧養關懷」提醒（不是懲罰，是提醒同工去關心會友近況），觸發時機是：
  - P2 會友連續 4 次沒來
  - 短期行動不便資格到了追蹤日期
  - 幼兒同行資格接近孩子入學年（系統依生日與台灣學年制 9/1 切點自動算出到期日，細節見交接文件，Admin 不需自己換算）

這些提醒都在 [牧養關懷](#牧養關懷-adminpastoral) 頁處理。

### 1.6 系統什麼時候會發 LINE 通知

| 情境 | 通知內容 | 誰觸發 |
|---|---|---|
| 候補遞補，需 2 小時內確認 | 「有一個車位釋出給您，請 2 小時內確認」 | 系統自動 |
| 保留車位釋出（資訊性） | reservation_released | 系統自動 |
| 預約被取消（含晚期取消轉釋出） | reservation_cancelled | 系統自動 |
| 主日 10:20 P2 到場提醒 | 「您保留的車位將於 10:45 釋出，正路上請回覆」 | 系統自動 |
| 同工在現場按「請車主移車」 | move_car_request（A 一般／B 緊急／C 散會後／D 車位調度，四種文案任選） | Staff 現場操作 |

所有通知都是**教會 LINE 官方帳號（OA）代發**、不會外露同工或其他會友的個人聯絡方式；前提是該會友已完成 LINE 綁定（見 [綁定審核](#綁定審核-adminbindings)）。通知的排隊與送出機制、健康度監控見 [營運狀態](#營運狀態-adminops) 頁與 [dispatcher-ops.md](dispatcher-ops.md)。

## 二、Admin 後台每一頁在做什麼

登入網址：`/admin`。左側選單（手機版在頂端橫向捲動）共 11 個功能，以下依「會友資料 → 車位/現場 → 監控/稽核」大致分組：

### 會友管理 `/admin/members`
搜尋會友（姓名／電話／車牌）、看完整明細（電話、車牌、資格狀態），對**未綁定 LINE** 的會友產生一次性綁定碼交給本人。列表搜尋結果電話會遮罩，明細頁才顯示完整資料。

### 名單匯入 `/admin/import`
把教會提供的會友 CSV（一般名單或 P2 資格申請表）上傳、預覽、確認寫入。**這是給非技術人員的頁面**，完整逐步說明＋每種錯誤訊息的意思，見 **[member-import-ops.md](member-import-ops.md)**。
> 匯入只建立會友資料，**不會**讓會友收到通知——收得到通知需要另外走下面的「綁定審核」。

### 綁定審核 `/admin/bindings`
把「已知會友」的 LINE 帳號正式對應到系統裡的人（寫入 `line_id`），之後移車/通知才送得到本人。兩種申請來源：會友自己在 LIFF 填「姓名＋手機」，或用同工發的一次性綁定碼。**一律人工核對後才核准**。完整流程＋各種拒絕原因的意思，見 **[binding-ops.md](binding-ops.md)**。

### 資格審查 `/admin/eligibility`
列出「已過期」「待覆核」「60 天內到期」的 P2 資格會友，點進會友明細頁可以核准、撤銷、或標記已覆核（並設定下次追蹤日期）。**「標記已覆核」之後，之後的 CSV 匯入不會再覆蓋你的決定**——這是刻意設計，避免幹事人工審過的資格被下一批名單匯入悄悄推翻。

### 車位設定 `/admin/capacity`
設定當週／下週（**只開放這兩週，不能設更遠的未來**）的「總車位」與「保留・停用車位」數字，畫面會即時算出「可分配車位」給你確認。系統會擋掉會讓車位變負或低於已核准人數的設定；若你剛好跟另一位同工同時在改，畫面會提示「已過期，請重新整理」而不是悄悄互相覆蓋。

### 列印點名表 `/admin/print`
列印當週點名表紙本，作為現場網路異常時的備援（Staff 頁本身是「唯讀快取」網路異常也能撐一陣子，但徹底離線時紙本是最後防線）。這是主日前辦公室的準備工作，不是主日當天在地下室操作的頁面。

### 現場 PIN 管理 `/admin/staff-pin`
幫當週／下週的 Staff（現場停車服事同工）發放登入 PIN 碼（6 碼數字，只顯示一次）。PIN 有效期會涵蓋到該主日結束，所以可以提前幾天先發。同一週重新發一組新 PIN，舊的會立刻失效。

### 帳號管理 `/admin/accounts`
管理**其他** Admin 帳號（不能對自己操作）：停用／重啟／重設密碼／強制登出所有裝置。**新帳號的建立不在這頁**，需要開發者/技術人員用指令建立，見 [admin-account-ops.md](admin-account-ops.md)。系統保證永遠至少留一位啟用中的 Admin，不會被誤操作歸零。

### 營運狀態 `/admin/ops`
看 LINE 通知佇列的健康度（有幾筆等待送出、有幾筆失敗、失敗原因），失敗過多時可以「死信重送」（先預覽、確認後才真的重送）。細節見 **[dispatcher-ops.md](dispatcher-ops.md)**。

### 牧養關懷 `/admin/pastoral`
處理系統自動開立的關懷提醒（P2 連續未到／資格到期追蹤）。處理時可選擇是否同時把「連續未到次數」歸零——這是關心不是懲罰，違規分數本身這裡不會被動到。

### 稽核記錄 `/admin/audit`
唯讀查看系統裡誰在什麼時候做了什麼異動（例如誰停用了哪個帳號、誰改了車位設定）。目前保存期是 24 個月，到期後由排程作業清除（見 [go-live-checklist.md](go-live-checklist.md) 1.5 的稽核清除 cron）。

## 三、日常維運節奏建議

| 頻率 | 該做的事 | 對應頁面 |
|---|---|---|
| 每主日前（週間） | 確認車位設定沒問題；如有新申請資格要核准，走資格審查 | 車位設定、資格審查 |
| 每主日 | 視需要提前發放 Staff PIN | 現場 PIN 管理 |
| 每主日後 | 看一次「營運狀態」有沒有異常失敗；有新綁定申請就審核 | 營運狀態、綁定審核 |
| 每次教會提供新名單 | 走匯入流程 | 名單匯入 |
| 每月 | 檢查稽核記錄清除 cron 有沒有正常跑（見 go-live-checklist 1.5） | 稽核記錄 |
| 不定期 | 有牧養關懷提醒就處理 | 牧養關懷 |

## 四、試營運怎麼做

> **這是目前教會最需要的部分**：開發已經全部完成，接下來是把真實會友資料、真實 LINE 官方帳號一步步接上，確認沒問題再全面開放。**唯一權威的順序清單是 [go-live-checklist.md](go-live-checklist.md)**——本節是給 admin 看的白話說明，實際勾選、驗證方式一律以該文件為準。

試營運的核心精神：**先讓少數人走完整流程、觀察至少一個主日沒出狀況，再逐步擴大**，而不是一次對全教會開放。

### 目前狀態（截至本文件撰寫時）

依 [go-live-checklist.md](go-live-checklist.md)：備份、正式 OA 接線、文案 sign-off 已完成。**接下來要做的是**：匯入真會友名單 → 排程/稽核清除上線確認 → 開啟真實送出 → 分批放行。實際勾選狀態請直接看該文件，不要以本文件的敘述為準（會隨時間過期）。

### Step 1：匯入真會友名單（1.3）
教會辦公室提供 CSV → Admin 走 [名單匯入](#名單匯入-adminimport) 頁 preview → 確認寫入。此時資料進系統了，但還沒人收得到 LINE 通知（因為還沒綁定）。

### Step 2：排程與稽核清除確認（1.5）
確認 11 個通知排程任務都正常指到正式網址，並新增每月一次的稽核記錄清除排程——`/admin/audit` 頁對外宣稱「24 個月後清除」，這句話要等這個排程真的在跑才算數。這步驟主要是技術/排程操作者（Scheduler operator）的工作。

### Step 3：開啟真實送出（1.6）
把系統從「模擬送出」切成「真的透過 LINE 送出」。切換前會先對**一位知情的操作者帳號**送一則測試通知，確認真的收得到，才繼續。

### Step 4：分批放行（1.7，最關鍵的一步）
1. **先找一個小組**（例如同工自己這一組），走綁定流程：由 Admin 發一次性綁定碼給組員 → 組員在 LINE 傳「綁定 `<碼>`」→ Admin 到 [綁定審核](#綁定審核-adminbindings) 頁核准。
2. **只對這個小組開放真實送出**（其他人維持不送，避免大範圍出錯）。
3. **觀察至少一個完整主日循環**：看 [營運狀態](#營運狀態-adminops) 頁的健康度——沒有異常失敗、沒有卡住的處理中列、未綁定的車主畫面上有正確顯示「無法通知」的提示。
4. **確認沒問題後才擴大到下一批**，重複「發碼綁定 → 觀察一個主日 → 再擴大」直到全教會。

同時，[oa-onboarding-and-move-car-copy.md](oa-onboarding-and-move-car-copy.md) 有教會端「怎麼提高加入官方帳號比例」的具體做法（歡迎訊息、圖文選單、主日現場 QR code 等），這是讓分批放行能順利擴大的前提——沒加入 OA 的人，發了綁定碼也綁不了。

### 什麼時候算「正式營運」

當全教會的停車會友都完成綁定、真實送出穩定運作一段時間、[go-live-checklist.md](go-live-checklist.md) 全部項目打勾，就算正式營運，之後轉為第三節的「日常維運節奏」。

## 五、出狀況怎麼辦

- **通知送不出去 / 大量失敗**：先看 [營運狀態](#營運狀態-adminops) 頁健康度找原因（常見是會友沒綁定 LINE）；需要緊急煞車時，停送通知的操作屬於技術/排程操作者（Scheduler operator）權限，見 [go-live-checklist.md](go-live-checklist.md) 第 2 節 Rollback、[dispatcher-ops.md](dispatcher-ops.md)。
- **所有 Admin 帳號都被誤停用、沒人能登入**：這是設計上不該發生的狀態（系統會擋「停用最後一位」），萬一真的發生需要技術人員直接操作資料庫救援，見 [admin-account-ops.md](admin-account-ops.md)「真的卡住時」。
- **匯入中途出錯**：不用手動清資料，保留原始 CSV，重新跑一次「上傳並預覽 → 確認寫入」即可，系統設計成不會重複建立同一個人，見 [member-import-ops.md](member-import-ops.md) 第 4 節。
- **看不懂的錯誤訊息 / 結果跟預期差很多**：不要直接送出／確認，先截圖（只在你與開發者之間，不外流）聯絡開發者。
