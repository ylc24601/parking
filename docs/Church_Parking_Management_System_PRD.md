# ⛪ 教會主日停車管理系統 (MVP) 產品需求文件 (PRD)

## 一、 專案背景與核心目標
* **資源現況：** 教會地下室共有 23 個固定編號停車位，但每週主日約有 40–50 台車有停車需求。
* **核心目標：** 建立一套落實「恩慈關懷、公平透明、資源共享」原則的停車分配系統。透過 LINE 自動化管理，將車位優先保留給有特殊需求與長幼家庭的會友；並引入輪替機制與恩慈取消設計，降低服事同工負擔，同時鼓勵會友準時聚會。
* **使用場景：** 具備 Wi-Fi 網路的主日地下室停車場。

---

## 二、 MVP 範圍與車位管理原則
1. **可用車位公式化：** 當週**公開可分配數** = 總數 (23) - 外賓保留位 (guest_reserved) - 當週停用/異常車位 (blocked spaces) - **當週仍占用名額的全職同工數 (active_full_time_staff_reserved)**。
   * 全職同工 (P1) 與外賓皆**不進入公開排序**，於分配前先扣除。
   * `active_full_time_staff_reserved = 全職同工名單人數 - 本週標記「不停車」人數`。
2. **不指定實體車格：** MVP 階段僅管理當週可用車位總數，不自動指定實體車位編號。實際停放位置由現場 Staff 依地下室動線與現場狀況引導。

---

## 三、 使用者角色與權限 (RBAC)
系統採「資料最小化」與「權限隔離」原則，保護會友隱私：

1. **一般會友 (User)**
   * 透過教會 LINE 官方帳號 (LIFF 網頁) 操作。
   * 綁定資料、登錄車牌、每週宣告長幼同行、預約與取消。
2. **停車管理同工 (Staff)**
   * 當週主日在地下室第一線引導的停車管理同工，使用手機或平板操作（停車場無電腦）。
   * 使用極簡版網頁，僅能檢視當週清單、執行「到場點名 / 遲到補點名」、登記「現場散客 (Walk-in)」、點擊「結束當週點名」。
   * **隱私隔離：** 介面僅顯示姓名、車牌與通用的「⭐ 優先車位」圖示，**不可見**會友的具體 P1/P2 原因（行動不便、孕婦、年長/幼兒等）與違規分數。
3. **管理員 (Admin)**
   * 教會辦公室幹事或系統管理者。
   * 管理 P1 全職同工名單、審核 P2 關懷資格、檢視牧養關懷提醒、手動保留特定車位給外賓、手動補結算點名、匯出數據。操作皆需留存稽核軌跡 (Audit Logs)。

---

## 四、 車輛與預約限制防護
1. **一帳號一預約：** 每個 LINE 帳號每週僅能持有一筆有效申請。
2. **多車登錄彈性：** 一個帳號可綁定多台車牌，每週預約時動態選擇「本週開哪台車」。
3. **車牌唯一性：** 同一組車牌不可同時綁定於多個 LINE 帳號。若需轉移，需由 Admin 協助。
4. **臨時換車機制：** 會友於主日抵達前可自行在系統修改預約車牌；若抵達現場才換車，可由 Staff 於後台加上「現場換車備註」。

---

## 五、 核心排序規則（輪替公平制）
預約截止後，系統不採先搶先贏，而是以「關懷優先」為第一層、「輪替公平」為第二層進行自動排序：

1. **關懷優先級（Effective Priority）：**
   * **P1 (全職同工 / 外賓)：**
     * **全職同工：** 由 Admin 直接維護名單，享有完全保留名額（不佔公開分配車位數）。若當週在外教會服事，**須於週五前在系統標記「本週不停車」**，名額自動釋出給一般候補。
     * **外賓：** 由 Admin 每週手動設定保留車位數，不進入公開排序。
   * **P2 (關懷優先)：** 具備 Admin 核准之 P2 資格，參與公開車位排序且排序最前。P2 資格類型如下：
     * **長期行動不便 / 身心障礙需求：** 經本人說明或牧長/幹事確認後核准（**不需上傳身障手冊或診斷證明**），每週自動套用、無需每週宣告，資格長期有效直到 Admin 手動取消。
     * **短期行動不便：** 同享 P2 優先排序，每週自動套用，無需宣告。Admin 核准時可設定「下次追蹤日期」（不設則預設核准日起 4 週後）。到期時系統於 Admin 後台產生追蹤提醒，由 Admin 確認是否繼續保留或取消資格。
     * **孕婦：** 經 Admin 核准後套用，**有效期至預產期後 3–6 個月**；到期後系統提醒 Admin 關懷確認。若產後仍有幼兒同行需求，由 Admin 轉為「幼兒同行」資格（避免會友產後多年系統仍顯示孕婦狀態）。
     * **年長者同行：** 具備 P2 資格，**且當週宣告**有 75 歲以上年長者同行；當週未宣告則以 P3 排序。Admin 核准時記錄長者姓名與生日，供資格驗證及年齡確認。
     * **幼兒同行：** 具備 P2 資格，**且當週宣告**有學齡前幼兒同行；當週未宣告則以 P3 排序。Admin 核准時記錄孩童姓名與生日；系統依生日自動於孩子達學齡前發出 Admin 資格到期提醒。
   * **P3 (一般會友)：** 依違規分數 (Penalty Score 0–3) 高低排序，分數越高排序越後；分數相同者，再比輪替公平日期。
2. **輪替公平（最後成功停車日期）：**
   同一個優先級內，比對「最後一次成功停到車的日期 (`last_successful_attended_at`)」。從未停過者 > 距離上次停車越久者。
3. **同分判定（申請時間）：**
   若上述皆相同，才比對當週送出申請的時間，先申請者得。

## 六、 預約時程、分配與遞補機制
1. **週一 09:00（開放申請）：** 開放當週主日預約，狀態皆為 `pending`。
2. **週五 18:00（自動分配）：** 系統關閉排隊，依「輪替公平制」排序。名額內轉為 `approved` (已核准)，其餘列入 `waiting` (候補)。
3. **晚鳥即時預約：** 週五分配後若未滿額，系統保持開放，新申請採「隨送隨核准」直到滿額。
4. **候補動態遞補與鎖位 (temp_approved)：**
   若有人取消預約，系統對 Waiting #1 執行遞補，遞補期間**該車位暫時鎖定**，不開放晚鳥或下一順位取得：
   * **週五 18:00 - 週六 23:59 取消：** 啟動「緩衝確認制」。系統發送 LINE 通知，候補者需於 **2 小時內**確認保留。確認轉 `approved`；逾時未確認（`offer_status='expired'`）或主動放棄（`offer_status='declined'`）則釋出給下一位。**遞補失敗者主狀態維持 `waiting`、保留原排位，等待下次遞補機會。**
   * **週日 00:00 後取消：** 啟動「即時核准制」。跳過確認，直接將 Waiting #1 轉為 `approved`，並發送保留至 10:30 的即時通知。
   * **邊界情況（週六 23:xx 取消）：** 若週六 23:00 後取消觸發 `temp_approved`，系統將 `offer_expires_at` 設為 `MIN(發送時間 + 2小時, 週日 00:00)`。週日 00:00 的排程任務會掃描所有仍處於 `temp_approved` 狀態的紀錄，自動升格為 `approved`，補發即時通知，無需候補者手動確認。

---

## 七、 取消與恩慈防護網
1. **早期取消 (週五 18:00 前)：** 無負面紀錄 (`cancelled_by_user`)。
2. **晚期取消 (主日 10:30 前)：** 遇突發狀況可隨時取消，**不計違規分數** (`cancelled_late`)，並立即觸發遞補。

---

## 八、 主日現場點名與釋出機制
1. **現場點名（10:00 - 10:30）：** Staff 於後台一鍵勾選入場車輛為 `attended`。
2. **10:20 P2 到場提醒：** 系統發送 LINE 通知給所有 P2 已核准但尚未到場的會友，提示車位保留至 10:45，可點選「正在路上」延緩釋出。
3. **10:30 一般會友逾時釋出：**
   * 10:30 準時將 **P3 一般會友**未到場的保留位轉為逾時釋出 (`released_late`)。
   * 同時廣播通知候補者：*「目前地下室有臨時釋出車位，候補與現場車輛可至入口詢問同工，依現場順序停放，額滿為止。本通知不代表保證保留車位，請勿違停影響動線。」*
4. **10:45 P2 逾時釋出：**
   * **P2 會友**保留至 10:45。10:45 仍未到場**且未回覆「正在路上」**者，轉為 `released_late`。
   * **「正在路上」採有界寬限：** 已回覆者延長至 **10:55（grace_deadline）**；10:55 仍未到場即釋出。**不無限延長**——因 10:30 後已進入現場調度，無限保留與下款矛盾且無法稽核。
   * **判定一致性：** 每筆預約以其釋出截止時間 (`release_deadline_at`：P3=10:30、P2=10:45、P2 且正在路上=10:55) 為界。在截止前到場一律算 `attended`（例如 P2 於 10:35 到場為 `attended`，非遲到補點名）。
   * **10:30 後現場最大：** 系統進入現場調度階段，**不再保證依候補順位保留車位**，以現場交通安全、入口秩序與 Staff 指引為優先。
5. **遲到彈性：** 10:30 後抵達者，若 Staff 尚未撤崗，仍可補點名 (`attended_after_release`)，免除違規（但不保證有實體車位可停）。
6. **防呆提醒機制：** 若 Staff 忘記結算，12:30 發送提醒給 Staff；14:00 仍未結束，通知 Admin 手動補結算，避免系統卡死。

---

## 九、 未到場 (No-show) 結算與後台牧養關懷
Staff (或 Admin) 點擊「結束當週點名」時，系統針對整場未出現且未取消 (`released_late`) 的預約進行結算 (`no_show`)：

1. **一般會友 (P3) 降級與恢復：**
   * **降級：** 發生一次 No-show，`penalty_score` 加 1（上限為 3），下次排序後退。**前端不顯示 P4/P5**，僅溫和提示「因過往紀錄排序暫時調整」。
   * **恢復：** 未來只要成功預約且到場，`penalty_score` 減 1，直至歸零。Admin 亦可針對特殊情況手動歸零。
2. **恩慈豁免與關懷提示 (P2)：**
   * **免罰：** P2（行動不便、孕婦、年長/幼兒同行）預約未到，不增加 `penalty_score`，下週保有 P2 優先權。
   * **關懷提示：** Admin 後台的「牧養關懷提醒」由以下三種情形觸發，性質皆為關心而非懲罰：
     1. **連續未到：** P2 會友連續 4 次預約未到，建議牧長/幹事主動聯繫了解家庭狀況。P1 全職同工若未到係因外教會服事，應事先標記「本週不停」，不納入此統計。
     2. **短期行動不便追蹤到期：** Admin 設定之追蹤日期到期（預設核准日起 4 週），提醒 Admin 確認會友狀況是否改善，決定繼續保留或取消 P2 資格。
     3. **幼兒資格即將到期：** 依孩童生日計算，孩子接近學齡（入學年份前一個月）時提醒 Admin 審核是否取消幼兒同行資格。
   * **重置：** P2 成功到場（`attended` 或 `attended_after_release`）後，`consecutive_no_show` 自動歸零；Admin 亦可針對特殊情況手動歸零，並寫入稽核軌跡。

## 十、 系統狀態機對照表 (Reservation Status Enum)

| 狀態代碼 | 狀態名稱 | 實務意義與觸發時機 |
| :--- | :--- | :--- |
| `pending` | 排隊中 | 週一至週五 18:00 間的初始申請。 |
| `approved` | 已核准 | 分配成功或成功遞補，保留車位至 10:30。 |
| `temp_approved`| 遞補待確認 | 週六遞補過渡狀態，鎖定名額等待 2 小時確認（＝「offer 進行中」）。 |
| `waiting` | 候補中 | 分配後未能取得車位的名單。遞補失敗者主狀態維持於此（見下方 `offer_status`）。 |
| `attended` | 已到場 | 在該預約 `release_deadline_at` 前抵達並完成點名（P3 ≤10:30、P2 ≤10:45、P2 正在路上 ≤10:55）。 |
| `released_late` | 逾時釋出 | 未到場者自動釋出保留權。**P3 於 10:30 觸發；P2 於 10:45 觸發；P2 已回覆「正在路上」於 10:55 觸發。** |
| `attended_after_release` | 遲到補點名 | 超過該預約 `release_deadline_at` 後才抵達，經 Staff 補點名（免除違規）。 |
| `no_show` | 完全未到 | **結算時觸發**。整場未出現且未取消。 |
| `cancelled_by_user`| 早期取消 | 週五 18:00 前自行取消（無負面紀錄）。 |
| `cancelled_late`| 晚期取消 | 主日 10:30 前自行取消（立刻遞補，無違規紀錄）。 |
| `walk_in` | 現場散客 | 無預約但直接進場，由 Staff 手動新增以利空位計算。 |

*(註：停用車位或活動關閉等場地狀態，將以 `Weekly_Events` 表格中的 `blocked_spaces` 或 `status` 處理，非個人預約狀態)*

> **遞補結果以子欄位 `offer_status` 表達，不再作為主狀態：** `null`（無遞補經歷）/ `expired`（2 小時未確認）/ `declined`（主動放棄）。遞補失敗時主狀態維持 `waiting`，僅更新 `offer_status`，以確保「維持原排位」與主狀態不矛盾（原 `offer_expired` / `offer_declined` 主狀態已移除）。

## 十一、 個資保護與隱私安全
1. **使用者同意與用途：** 首次註冊需明確同意蒐集車牌與 LINE ID 供停車分配及通知使用，不作其他用途。
2. **資料最小化：** P1/P2 資格僅存為 Boolean 加資格類型；幼兒同行與年長者同行另記錄依附人員姓名與生日（用於資格驗證與到期提醒）。系統不索取、不記錄任何診斷證明或身障手冊等敏感個資。
3. **資料保存與刪除：** * 停車預約紀錄保存 1–2 年。
   * Audit Logs 保存 2–3 年。
   * 會友可向 Admin 申請刪除車牌與停用帳號。
4. **安全稽核 (Audit Logs)：** 凡 Admin 核准資格、手動修改狀態等權限行為，皆自動寫入稽核軌跡。

## 十二、 建議開發順序 (Execution Phases)
強烈建議以漸進方式驗證高風險邏輯：
1. **Phase 0: Algorithm & State Machine Mocking (TDD)**
   先撰寫腳本產生假資料，純測試週五演算法、候補鎖位(`temp_approved`)機制、10:30 釋出與降級恢復邏輯。
2. **Phase 1: Database & Constraints**
   建立 Supabase Schema、外鍵約束、唯一性限制 (Unique License Plates)、Migrations。
3. **Phase 2: Core API Handlers**
   實作週五 Cron Job、2小時候補機制、取消流程、10:30 釋出與結算 API。
4. **Phase 3: Staff Dashboard**
   優先開發主日現場最重要的點名介面 (包含補點名、登記 Walk-in、結束點名與防呆機制)。
5. **Phase 4: Admin Dashboard**
   開發資格審核、設定每週車位數、手動保留車位與稽核檢視。
6. **Phase 5: LINE Bot & LIFF Frontend**
   會友端註冊綁定、車牌管理（含多車牌）、當週 P2 宣告（強制步驟）、申請/取消/換車介面。
   > ⚠️ **建議：** 基本 LIFF（登錄車牌、當週申請/取消）應於 Phase 3–4 間**平行開發**，讓 Staff/Admin Dashboard 開發期間能使用真實申請資料驗證流程邏輯，而非全部推遲至 Phase 5。
7. **Phase 6: Refinement & Reporting**
   優化 LINE 訊息文案、自動排程通知、開發**停車樣態分析**（車位使用率趨勢、No-show 率、候補平均等待週數、P1/P2/P3 分配結構）與報表匯出功能。
